Detail Specification
1.1
Project Description
Ho Chi Minh City University of Technology serves a large and diverse popula-
tion of daily parking users, including undergraduate students, graduate students,
doctoral candidates, lecturers, staff members, and short-term visitors. As vehicle
density increases while parking capacity remains limited, the current parking process
becomes less efficient at several points. Common issues include queues at entry and
exit gates, underutilized parking areas caused by poor visibility of available slots,
inconsistent handling of exceptional cases, and fragmented fee collection workflows.
The proposed Smart Parking System aims to address these limitations by build-
ing a campus-scale parking platform that combines software services with IoT-based
sensing infrastructure. Instead of treating parking as a purely manual gate-control
activity, the system provides coordinated support for user authentication, parking-
session management, occupancy tracking, operational guidance, and billing. The
platform is expected to function in a realistic environment where many requests
may occur at the same time, external services may respond slowly, and data from
devices may occasionally become delayed or unreliable.
A major characteristic of the project is the coexistence of multiple user cate-
gories with different rules. University members can use their institutional identity
to access the parking service, while visitors and users in exceptional situations may
still need temporary access without going through the normal centralized account
flow. In parallel, administrators and operators must be able to supervise operations,
configure policies, correct exceptional cases, and inspect logs for traceability. There-
fore, the system is not only a user-facing application but also an operational tool
for campus management.
Another essential aspect is real-time parking visibility.
Each parking slot is
associated with an occupancy sensor, and slot information is forwarded through
IoT gateways to the central system. Based on these updates, the system maintains
a near real-time representation of available capacity and supports traffic guidance
inside the campus. Guidance may be delivered through mobile/web interfaces or

its available capacity.

UI Design
4.1
Design Rationale and User-flow Logic
The UI design is organized around the main operational problems identified in
the campus parking context: congestion at entry and exit points, limited real-time
visibility of parking availability, difficulty handling visitors or exceptional no-card
cases, fragmented billing procedures, and the need for operator/admin supervision.
To address these issues, the interface is structured into six solution-oriented modules:
Authentication and User Access, Check-in & Temporary Entry, Slot Monitoring and
Availability, Guidance and Session Closure, Billing and Payment, and Administra-
tion, Policy and Audit.
This structure is intended to reduce cognitive load for each user group by present-
ing only the decisions relevant to their role and current task. University members are
guided through authenticated access and parking usage, visitors are guided through

Faculty of Computer Science and Engineering
temporary entry and payment, while operators and administrators are supported
through monitoring, exception handling, logging, and policy control. In this way,
the UI is not designed merely as a set of screens, but as an interface layer that
operationalizes the project’s core requirements and constraints.

University Member Flow. This flow supports authenticated campus users who
access parking through institutional identity. It begins with HCMUT_SSO-based
authentication, continues with visibility of current parking availability, then supports
check-in, active parking-session tracking, and check-out. The final billing/status
step reflects the requirement that some user groups may be charged according to
role-based or billing-period policies rather than only immediate payment.

directly addresses the requirement for temporary access independent of centralized
authentication.
Administrator / Operator Flow. This flow supports monitoring and governance
tasks rather than ordinary parking usage. Instead of focusing on entry, it focuses on
configuration, operational correction, logs, audit records, and alerts. This reflects
the requirement that the system must support different operational responsibilities,
including system administrators and parking operators.

8
Architecture Design
8.1
Design Rationale
The Smart Parking System is designed as a modular and integration-oriented
platform because it must serve multiple user categories, coordinate real-time parking
operations, and interact with several external services and device layers. In addition
to ordinary user-facing functions, the system must support operational supervision,
abnormal-session recovery, billing synchronization, and policy governance. There-
fore, the architecture is organized to separate presentation concerns, application
logic, domain entities, external integrations, and persistence responsibilities.
The architectural design follows a layered and component-based approach in
order to improve maintainability, traceability, and future extensibility. This sep-
aration is especially important because the system integrates with HCMUT_SSO
for authentication, HCMUT_DATACORE for institutional profile data, BKPay for
payment processing, IoT gateways for slot-status updates, and electronic signage for
parking guidance. By isolating such dependencies through dedicated adapters and
service components, the system can remain operational even when external services
are delayed, unavailable, or partially degraded.
The architecture also reflects the operational characteristics identified in pre-
vious submissions.
University members, visitors, parking operators, and system
administrators do not interact with the same interface or follow the same workflow.
Similarly, normal flows and exception-oriented flows, such as denied entry, unknown
slot state, abnormal-session correction, or payment synchronization delay, require
different responsibilities across modules. As a result, the architecture is structured
not only for functional completeness, but also for realistic execution in a campus
parking environment.

Module Overview

10
Bonus Test Cases
10.1
Overview
This section is optional and included only as a bonus. The purpose of the test-
case section is to demonstrate that the architectural and structural design of the
Smart Parking System can be validated through representative execution scenarios.
The following test cases focus on critical and high-risk flows, especially those in-
volving institutional integration, temporary access, real-time monitoring, abnormal
session handling, and payment synchronization.

Objective
Verify that a university member can authenticate

Pre-condition
HCMUT_SSO and HCMUT_DATACORE are available.

1. User submits login request. 2. System authenticates
through HCMUT_SSO. 3. System retrieves profile from
HCMUT_DATACORE. 4. System evaluates parking
privilege.

Access is granted or limited according to active policy, and
the access decision is recorded.

Field
Description / Content
Test Case ID
TC_ENTRY_01
Objective
Verify that a visitor can receive temporary access and start
a parking session.
Pre-condition
Gate-side interface is operational. Temporary-ticket
handling is enabled.
Test Steps
1. Visitor requests entry. 2. System classifies request as
temporary-entry case. 3. System issues a temporary ticket.
4. System assigns a slot or zone. 5. System creates a new
session.
Expected Result
A temporary ticket is issued and a new parking session is
created successfully.

Field
Description / Content
Test Case ID
TC_MON_01
Objective
Verify that the system explicitly marks a slot as unknown
when reliable updates are missing.
Pre-condition
A slot exists and no valid sensor update has been received
within the configured timeout window.
Test Steps
1. Monitoring logic detects stale or missing sensor data. 2.
System updates the slot state. 3. System logs the anomaly.
Expected Result
The slot is marked as Unknown and the anomaly is
recorded for monitoring.
